## Vision

### 10x Check
A **Race-Day Operations Console** — a single surface where judges and the head
organizer can see and manage everything that goes wrong during live competition,
with the system automatically protecting data integrity. Scratch cascade is just
one operation in a broader "competitor status change" system. Relay payouts
integrated into a unified payout dashboard. Live integrity monitor surfaces
orphaned state warnings in real-time.

### Platonic Ideal
Judge taps "Scratch." System previews every downstream effect in plain English.
Judge confirms (step-by-step, each effect individually). Everything cascades
transactionally per step. Audit trail records who, when, what. Head organizer's
dashboard shows real-time feed. Undo available within window. At awards, payout
dashboard shows all earnings with settlement tracking.

## Key Implementation Decisions
- **Cascade UX:** Atomic with preview opt-out. Preview modal shows all downstream effects
  with checkboxes (all checked by default). Judge unchecks effects to skip. On confirm,
  all checked effects execute in ONE `db.session.begin_nested()` savepoint. If any step
  fails, the entire cascade rolls back. No partial state.
- **Relay payouts:** Per-team lump sum. Organizer enters dollar amounts per placement
  (1st, 2nd, 3rd, etc.) on a relay-specific payout config page. Does NOT use
  `PayoutTemplate` model (that's per-position for individuals). Stored in
  `Event.payouts` JSON (now freed from relay state) as `{"1": 200, "2": 100}`.
- **Scratch cascade architecture:** Two-phase:
  1. `compute_scratch_effects(competitor, tournament)` returns `List[CascadeEffect]`
     where each effect has `description`, `effect_type` (enum: 'event_result',
     'partner', 'relay_team', 'standings'), and `affected_entity_id`. Pure
     computation, no DB writes. Effects are serializable (no callables stored).
  2. GET route calls `compute_scratch_effects()`, renders as checkboxes with
     `effect_type` + `affected_entity_id` as hidden form values.
  3. POST route receives checked effect types/IDs, calls
     `compute_scratch_effects()` again (fresh computation to avoid stale state),
     filters to only the checked effects, then calls `execute_effects(effects)`
     which runs ALL checked effects in a single outer savepoint. Atomicity
     guaranteed: if any effect fails, the entire cascade rolls back.
- **Both scratch entry points wired:** Both `routes/registration.py` (registration-level
  scratch) and `routes/scoring.py` (heat-level scratch) call `cascade_scratch()`.
  Inline status updates in registration.py are replaced with the service call.
- **Transaction composability:** `ProAmRelay._save_relay_data()` and
  `PartneredAxeThrow._save_state()` accept optional `commit=False` parameter.
  When called from cascade service, skip internal commit to participate in the
  outer transaction. Same pattern as `_rebuild_individual_points()` in scoring_engine.
- **Field separation:** New `event_state` TEXT column (JSON). Migrate via Alembic:
  for each Event where `name == 'Pro-Am Relay'`, copy `payouts` → `event_state`,
  then set `payouts = '{}'`. Reversible downgrade copies back.
- **Permissions:** Scratch cascade requires `is_judge` role (same as all management
  routes via `require_judge_for_management_routes` before_request hook in app.py).
- **Concurrency:** Each cascade step acquires a row-level lock on the affected entity
  (competitor, heat, relay team) via `with_for_update()`. If lock unavailable (another
  judge scratching simultaneously), step returns a conflict error surfaced in the UI.
  Note: `FOR UPDATE` is PostgreSQL-only; SQLite (dev/test) silently ignores it. This
  is acceptable — dev is single-user. Concurrency protection matters only in prod (PG).
- **Re-finalization trigger:** When a scratch cascade removes a result from a finalized
  event, that event's `is_finalized` is set to False (same pattern as edit-after-finalize
  in routes/scoring.py lines 360-362). The "pending re-finalization" badge makes this
  visible on the event card.

## Competitor Status State Machine

```
                    ┌─────────────┐
  Registration ───► │   active    │ ◄─── reinstate (deferred)
                    └──────┬──────┘
                           │
              ┌────────────┼─────────���──┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ scratched│ │   dns    │ │   dq     │
        └──────────┘ └──────────┘ └──────────┘

  EventResult.status: pending → completed | scratched | dnf | dns | dq
  
  Only 'completed' results contribute to standings/payouts.
  Scratch cascade sets Competitor.status='scratched' AND
  EventResult.status='scratched' on affected results.
```

Valid transitions for V1:
- `active → scratched` (via scratch cascade)
- `active → dns` (via heat scoring — competitor no-shows)
- `active → dq` (via judge action on heat)
- All other transitions (reinstatement, etc.) deferred.

### Settlement Tracking
New `payout_settled` Boolean column on `EventResult` (default False).
For relay: settlement tracked per-team in event_state JSON as

### Relay Team Health States
- **Green:** All 8 members have `Competitor.status='active'`.
- **Yellow:** 1-2 members scratched/inactive but team still meets minimum:
  at least 3 active pro members (with at least 1M and 1F pro) AND
  at least 3 active college members (with at least 1M and 1F college).
- **Red:** Fewer than 3 active members in either division (pro or college),
  OR any division has 0 active members of a required gender. Cannot race.

## Failure Modes

| Cascade Step | Failure Mode | Handling |
|---|---|---|
| Event result scratch | DB lock conflict | Savepoint rollback, UI shows "Result locked by another judge" |
| Partner scratch | Partner already scratched | Skip step (idempotent), log warning |
| Relay team removal | Team JSON parse error | Savepoint rollback, log error, UI shows "Relay update failed" |
| Standings recalc | _rebuild_individual_points fails | Savepoint rollback, standings show stale data with warning |
| Settlement toggle | Concurrent toggle | Optimistic lock via version_id on EventResult |
| Data migration | Relay JSON malformed | Skip row, log warning, leave in payouts field |

## Migration Safety

Migration `add_event_state_and_settlement.py`:
1. `op.add_column('events', Column('event_state', Text, nullable=True))`
2. `op.add_column('event_results', Column('payout_settled', Boolean, server_default='false'))`
3. Data migration (Python, not raw SQL): For each Event where `name == 'Pro-Am Relay'`,
   the current `payouts` field contains ONLY relay team JSON (no payout amounts exist
   yet — that's the gap this plan solves). So the migration is a direct copy:
   `event_state = payouts; payouts = '{}'`. No JSON parsing/splitting needed because
   relay events have never had payout configuration. If a future migration encounters
   mixed data, use a Python function to parse and split by key presence.
4. Reversible downgrade: `UPDATE events SET payouts = event_state WHERE event_state IS NOT NULL AND name = 'Pro-Am Relay'`; drop columns.
5. Railway `releaseCommand = "flask db upgrade"` handles automatic deployment.
6. Zero-downtime: both columns are additive (nullable new column, no drops).
